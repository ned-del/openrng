5.1 — Family Overview

5.1 — 家族概覽

The OpenRNG patent family is not a collection of independent inventions. It is a coordinated set of claims that, together, cover the complete Computational Trust Infrastructure — from entropy generation through governance. Each patent maps to exactly one layer of the Trust Stack. Each claims a distinct responsibility. Together, they form a defensible, non-overlapping intellectual property portfolio that mirrors the architecture it protects.

The analogy is Apple's chip family: the A-series, M-series, S-series, and U-series are not independent products — they are components of a unified computing platform, each optimized for its layer (mobile, desktop, wearable, spatial). Similarly, the OpenRNG patent family comprises five components of a unified trust platform.

OpenRNG 專利家族不是獨立發明的集合。它是一組協調的權利主張，共同涵蓋完整的計算信任基礎設施——從熵生成到治理。每件專利精確對應信任堆疊的一層。每件主張一個獨特的職責。它們共同形成一個可防禦的、不重疊的智慧財產權組合，與它所保護的架構相互映照。

類比是蘋果的晶片家族：A 系列、M 系列、S 系列和 U 系列不是獨立產品——它們是統一運算平台的組件，每個都為其層級（行動、桌面、穿戴、空間）最佳化。同樣，OpenRNG 專利家族包含統一信任平台的五個組件。

5.2 — Patent #1: Trust Source Layer (OpenRNG Core)

5.2 — 專利一號：信任來源層（OpenRNG 核心）

Original Filing: "Method and System for Gaming Random Number Generation"

New Positioning: The foundational Trust Source — the layer that produces high-quality, multi-source entropy as the raw material of Computational Trust.

Key Claims:

  • Verifiable Delay Function (VDF) seed generation for deterministic, time-locked entropy production
  • Deterministic batch generation: VDF seed → N leaf hashes organized in a Merkle tree
  • On-chain Merkle root anchoring via a single blockchain transaction per batch (up to 65,536 values)
  • Pool-based serving with sub-2ms latency: pre-generated entropy served from memory with Merkle proof inclusion
  • Multi-source entropy aggregation from independent providers (distributed beacons, blockchain hashes, hardware RNG)

Layer Mapping: Layer 1 of the Trust Stack

Technical Moat: The combination of VDF (time-locked, non-parallelizable computation), Merkle batch generation (one transaction proves thousands of values), and pool serving (sub-2ms latency) creates a unique architecture that is not covered by existing patents. Chainlink's VRF patent (US12401532B2) covers distributed oracle-based VRF; OpenRNG uses centralized VDF + Merkle batch — a structurally different architecture, different cryptographic primitive, and different trust model.

原始申請：「遊戲隨機數生成方法與系統」

新定位：基礎性的信任來源——產生高品質、多來源熵作為計算信任原始材料的層。

關鍵權利主張：

  • 可驗證延遲函數（VDF）種子生成，用於確定性、時間鎖定的熵生產
  • 確定性批次生成：VDF 種子 → N 個葉子雜湊組織在 Merkle 樹中
  • 透過每批次單一區塊鏈交易進行鏈上 Merkle 根錨定（最多 65,536 個值）
  • 基於池的服務，低於 2ms 延遲：從記憶體中提供預生成的熵，附帶 Merkle 證明
  • 來自獨立供應商的多來源熵聚合（分散式信標、區塊鏈雜湊、硬體 RNG）

層級對應：信任堆疊的第一層

技術護城河：VDF（時間鎖定、不可並行化計算）、Merkle 批次生成（一筆交易證明數千個值）和池服務（低於 2ms 延遲）的組合建立了獨特的架構，不被現有專利涵蓋。Chainlink 的 VRF 專利（US12401532B2）涵蓋基於分散式預言機的 VRF；OpenRNG 使用集中式 VDF + Merkle 批次——結構上不同的架構、不同的密碼學原語和不同的信任模型。

5.3 — Patent #2: Trust Authorization Layer (Lockbox)

5.3 — 專利二號：信任授權層（Lockbox）

New Positioning: Policy-gated access to trust primitives with cryptographic usage attestation.

Key Claims:

  • Policy engine for entropy consumption that evaluates requests against multi-dimensional constraints (quality floors, source requirements, rate limits, regulatory context)
  • Cryptographic usage attestation: every consumption event produces an immutable, signed record proving that a specific consumer used a specific trust primitive under specific policy constraints
  • Delegated authorization for multi-agent systems: a primary agent can delegate trust consumption rights to child agents within defined policy boundaries
  • Compliance-aware authorization: automatic mapping of consumer context to regulatory framework requirements (GLI-19, EU AI Act, HIPAA)

Layer Mapping: Layer 2 of the Trust Stack

Differentiation from OAuth/RBAC: Traditional authorization controls access to resources. Trust Authorization controls access to trust primitives and produces attestation of correct usage. The attestation is the innovation — not just granting access, but cryptographically recording that access was exercised correctly.

新定位：具有密碼學使用證明的策略閘門存取信任原語。

關鍵權利主張：

  • 用於熵消費的策略引擎，根據多維約束（品質下限、來源要求、速率限制、法規背景）評估請求
  • 密碼學使用證明：每個消費事件產生不可變的、已簽名的記錄，證明特定消費者在特定策略約束下使用了特定信任原語
  • 多代理系統的委託授權：主要代理可以在定義的策略邊界內將信任消費權利委託給子代理
  • 合規感知授權：自動將消費者背景對應到法規框架要求（GLI-19、EU AI Act、HIPAA）

層級對應：信任堆疊的第二層

與 OAuth/RBAC 的差異：傳統授權控制對資源的存取。信任授權控制對信任原語的存取，並產生正確使用的證明。證明是創新——不僅僅是授予存取權，而是密碼學地記錄存取權被正確行使。

5.4 — Patent #3: Trust Evidence Layer (VEO)

5.4 — 專利三號：信任證據層（VEO）

Original Filing: Verifiable Entropy Object provisional patent

New Positioning: The standard format for trust evidence — a self-describing, quality-scored, cryptographically signed, optionally blockchain-anchored, independently verifiable digital object.

Key Claims:

  • The VEO-1 object format: a structured digital object carrying entropy payload, source provenance array, aggregation metadata, confidence scoring, cryptographic proof, optional blockchain anchor, and lineage references
  • Entropy Confidence Score (ECS): a six-dimension weighted scoring system (freshness, diversity, independence, manipulation resistance, verification success, availability) producing a 0–1000 score with defined grade boundaries
  • Canonical signing protocol: deterministic transformation of a VEO into a signing payload through deep key sorting and stable serialization, enabling secp256k1 EIP-191 signatures verifiable by any party without network access
  • Five-level verification framework: from structural validity through cryptographic verification through blockchain-anchored verification, with distinct policy-failure and invalid states
  • Object classification system: VEO-1A (single-source), VEO-1B (composite), VEO-1C (anchored), VEO-1D (decision — reserved)

Layer Mapping: Layer 3 of the Trust Stack

Status: Provisional patent filed. VEO-1 v1.0 specification frozen.

Standards Path: VEO-1 is designed to become an open standard — analogous to JWT for identity claims. The object format, scoring system, signing protocol, and verification logic are fully documented and implementable by any party.

原始申請：可驗證熵物件臨時專利

新定位：信任證據的標準格式——自描述、品質評分、密碼學簽名、可選區塊鏈錨定、可獨立驗證的數位物件。

關鍵權利主張：

  • VEO-1 物件格式：攜帶熵載荷、來源出處陣列、聚合元數據、信心評分、密碼學證明、可選區塊鏈錨定和血統參考的結構化數位物件
  • 熵信心分數（ECS）：六維度加權評分系統（新鮮度、多樣性、獨立性、抗操縱性、驗證成功率、可用性），產生 0-1000 分數，有定義的等級邊界
  • 規範簽名協議：透過深層鍵排序和穩定序列化將 VEO 確定性地轉換為簽名載荷，實現任何方無需網路存取即可驗證的 secp256k1 EIP-191 簽名
  • 五級驗證框架：從結構有效性到密碼學驗證到區塊鏈錨定驗證，具有獨立的策略失敗和無效狀態
  • 物件分類系統：VEO-1A（單來源）、VEO-1B（複合）、VEO-1C（錨定）、VEO-1D（決策——保留）

層級對應：信任堆疊的第三層

狀態：臨時專利已申請。VEO-1 v1.0 規範已凍結。

標準化路徑：VEO-1 設計為成為開放標準——類似於 JWT 之於身份聲明。物件格式、評分系統、簽名協議和驗證邏輯都已完整記錄，任何方都可實作。

5.5 — Patent #4: Trust Coordination Layer

5.5 — 專利四號：信任協調層

Filing Status: New — to be filed

Positioning: How trust propagates between autonomous agents and across organizational boundaries.

Key Claims:

  • Trust chain construction: a protocol for linking child VEOs to parent VEOs via lineage references, creating directed acyclic graphs (DAGs) of trust that enable full decision-tree reconstruction
  • Multi-agent trust consensus: a method for combining individual VEOs from multiple autonomous agents into a jointly verifiable consensus VEO, where each participant's contribution is recorded and independently verifiable
  • Cross-system trust handoff: a protocol for packaging, transmitting, and verifying trust assertions across organizational trust boundaries without requiring mutual trust between organizations
  • Trust propagation invariants: enforcement of non-amplification (child trust ≤ parent trust), transparent degradation (reduced sources are recorded, not hidden), and traceability (every VEO in a chain is independently verifiable)

Layer Mapping: Layer 4 of the Trust Stack

Critical Distinction: Trust Coordination is a protocol, not an orchestrator. It does not control what agents do. It ensures that trust propagates correctly between them — analogous to TCP for data transmission. This distinction is critical for the patent claims: the invention is the trust propagation protocol, not agent orchestration.

Direction: Horizontal — peer-to-peer trust propagation between autonomous agents and systems.

申請狀態：新——待申請

定位：信任如何在自主代理之間和跨組織邊界傳播。

關鍵權利主張：

  • 信任鏈建構：透過血統參考將子 VEO 連結到父 VEO 的協議，建立信任的有向無環圖（DAG），實現完整決策樹重建
  • 多代理信任共識：將來自多個自主代理的個別 VEO 組合為可聯合驗證的共識 VEO 的方法，每個參與者的貢獻都被記錄且可獨立驗證
  • 跨系統信任交接：跨組織信任邊界封裝、傳輸和驗證信任斷言的協議，無需組織間相互信任
  • 信任傳播不變量：執行非放大（子信任 ≤ 父信任）、透明降級（減少的來源被記錄而非隱藏）和可追溯性（鏈中每個 VEO 都可獨立驗證）

層級對應：信任堆疊的第四層

關鍵區別：信任協調是協議，不是編排器。它不控制代理做什麼。它確保信任在它們之間正確傳播——類似於 TCP 之於資料傳輸。這個區別對專利權利主張至關重要：發明是信任傳播協議，不是代理編排。

方向：水平——自主代理和系統之間的點對點信任傳播。

5.6 — Patent #5: Trust Runtime

5.6 — 專利五號：信任運行時

Filing Status: New — to be filed

Positioning: Governance, compliance, and continuous verification for trust infrastructure.

Key Claims:

  • Runtime trust policy enforcement: continuous monitoring and enforcement of system-wide trust policies that cannot be evaluated at the per-request level (chain depth limits, source rotation, anchoring frequency, temporal consistency)
  • Compliance attestation generation: structured, machine-readable compliance attestations mapping trust infrastructure operations to regulatory framework requirements (EU AI Act, NIST AI RMF, GLI-19, ISO 42001)
  • Trust health monitoring: continuous computation of system-wide trust metrics (average ECS trends, source availability, verification rates, consumption patterns) with anomaly detection and alerting
  • Regulatory framework adapters: pluggable modules that map trust lifecycle operations to specific regulatory requirements, enabling automatic compliance reporting for multiple jurisdictions

Layer Mapping: Layer 5 of the Trust Stack

Boundary with Patent #4: The boundary is precise and intentional. Coordination (Patent #4) is horizontal — it governs trust flow between peers. Runtime (Patent #5) is vertical — it governs trust enforcement from top to bottom. Coordination is a protocol for trust transmission. Runtime is a governance engine for trust assurance. There is no overlap.

Direction: Vertical — policy enforcement and audit from top to bottom of the Trust Stack.

申請狀態：新——待申請

定位：信任基礎設施的治理、合規和持續驗證。

關鍵權利主張：

  • 運行時信任策略執行：持續監控和執行無法在逐請求層級評估的系統範圍信任策略（鏈深度限制、來源輪換、錨定頻率、時間一致性）
  • 合規證明生成：結構化、機器可讀的合規證明，將信任基礎設施操作對應到法規框架要求（EU AI Act、NIST AI RMF、GLI-19、ISO 42001）
  • 信任健康監控：持續計算系統範圍的信任指標（平均 ECS 趨勢、來源可用性、驗證率、消費模式），具有異常偵測和警報
  • 法規框架適配器：可插拔模組，將信任生命週期操作對應到特定法規要求，實現多個司法管轄區的自動合規報告

層級對應：信任堆疊的第五層

與專利四號的邊界：邊界是精確且有意的。協調（專利四號）是水平的——治理對等方之間的信任流動。運行時（專利五號）是垂直的——從上到下治理信任執行。協調是信任傳輸的協議。運行時是信任保證的治理引擎。沒有重疊。

方向：垂直——信任堆疊從上到下的策略執行和稽核。

5.8 — Family Coherence

5.8 — 家族一致性

The patent family is designed for coherence, not just coverage:

  • No overlap: Each patent claims a distinct layer. The boundaries between layers are architecturally defined and technically enforced. An IP challenge to one patent does not weaken the others.
  • Cross-references: Each patent references the others as related applications, establishing the family relationship and demonstrating the unified architecture.
  • Layered licensing: The non-overlapping structure enables flexible licensing — individual layer licensing (e.g., a gaming company licensing only Trust Source + Trust Evidence) or full-stack licensing for enterprise deployments.
  • Defensive depth: Even if one layer's claims are narrowed during prosecution, the remaining four layers maintain independent protection. The architecture is defensible at every level.

專利家族設計追求的是一致性，不僅僅是覆蓋範圍：

  • 無重疊：每件專利主張一個獨特的層。層之間的邊界在架構上定義且技術上強制執行。對一件專利的 IP 挑戰不會削弱其他專利。
  • 交叉引用：每件專利將其他專利引用為相關申請，建立家族關係並展示統一架構。
  • 分層授權：不重疊的結構實現靈活授權——個別層授權（如遊戲公司僅授權信任來源 + 信任證據）或企業部署的全堆疊授權。
  • 縱深防禦：即使一層的權利主張在審查過程中被縮窄，其餘四層維持獨立保護。架構在每個層級都是可防禦的。

6.1 — AI Agents (Primary Market)

6.1 — AI 代理（主要市場）

Problem: Autonomous AI agents make non-deterministic decisions — tool selection, response sampling, task routing, resource allocation — with no audit trail for the randomness involved. As agents gain authority over financial, medical, legal, and operational decisions, this becomes a liability for every organization deploying them.

Solution: Every agent decision backed by a VEO. Trust chains across multi-agent systems via the Coordination layer. Compliance attestations via the Runtime layer. SDK integration with major agent frameworks for one-line adoption.

Key Players: OpenAI, Anthropic, Google DeepMind, Microsoft, Cohere, LangChain, CrewAI, AutoGPT, Fixie

Entry Strategy: Open-source SDK + free tier → framework integration PRs → developer adoption → enterprise conversion. The trust lifecycle is invisible to the developer: openrng.getEntropy({ policy: 'ai-grade' }) — one line, full lifecycle.

問題：自主 AI 代理做出非確定性決策——工具選擇、回應抽樣、任務路由、資源分配——涉及的隨機性沒有稽核軌跡。隨著代理對金融、醫療、法律和營運決策獲得權限，這對每個部署它們的組織都成為一種責任。

解決方案：每個代理決策都由 VEO 支持。透過協調層實現多代理系統間的信任鏈。透過運行時層提供合規證明。與主要代理框架整合的 SDK，實現一行程式碼採用。

關鍵參與者：OpenAI、Anthropic、Google DeepMind、Microsoft、Cohere、LangChain、CrewAI、AutoGPT、Fixie

進入策略：開源 SDK + 免費層 → 框架整合 PR → 開發者採用 → 企業轉換。信任生命週期對開發者是透明的：openrng.getEntropy({ policy: 'ai-grade' })——一行程式碼，完整生命週期。

6.2 — Enterprise AI

6.2 — 企業 AI

Problem: Enterprise AI deployments face growing compliance requirements — SOX for financial decisions, GDPR for data processing, EU AI Act for high-risk systems. Current compliance relies on audit logs with circular trust dependencies. Enterprises need infrastructure-level trust assurance, not just documentation.

Solution: Trust Runtime provides automated compliance attestations. Trust Evidence (VEO) creates independently verifiable decision records. The full Trust Stack integrates with enterprise security infrastructure (IAM, SIEM, GRC platforms).

Key Players: AWS, Microsoft Azure, Google Cloud, Databricks, Snowflake, SAP, Oracle, Salesforce

Entry Strategy: Cloud marketplace listings → managed service → enterprise licensing with dedicated support. Target: Chief AI Officers and Chief Compliance Officers.

問題：企業 AI 部署面臨日益增長的合規要求——SOX 用於金融決策、GDPR 用於資料處理、EU AI Act 用於高風險系統。當前合規依賴具有循環信任依賴的稽核日誌。企業需要基礎設施層級的信任保證，而不僅僅是文件。

解決方案：信任運行時提供自動化合規證明。信任證據（VEO）建立可獨立驗證的決策記錄。完整信任堆疊與企業安全基礎設施（IAM、SIEM、GRC 平台）整合。

關鍵參與者：AWS、Microsoft Azure、Google Cloud、Databricks、Snowflake、SAP、Oracle、Salesforce

進入策略：雲端市場上架 → 託管服務 → 具專屬支援的企業授權。目標：首席 AI 長和首席合規長。

6.3 — Gaming & Gambling

6.3 — 遊戲與博弈

Problem: Gaming regulators (GLI, BMM, eCOGRA) require provably fair RNG. Current certification processes test the RNG implementation but rely on the operator's system to log and attest to correct operation — creating the same circular trust dependency found in audit logs.

Solution: VEO provides third-party verifiable proof of fair randomness for every game outcome. ECS scoring provides quantitative quality assurance. Blockchain anchoring provides temporal immutability. Any regulator can verify any outcome independently without trusting the operator.

Key Players: GLI (Gaming Laboratories International), BMM Testlabs, eCOGRA, major igaming operators, casino management system vendors

Entry Strategy: GLI-19 / GLI-33 certification → pilot with progressive operators → industry standard adoption.

The case for a new infrastructure category

新基礎設施類別的論證

7.1 — The Historical Pattern

7.1 — 歷史模式

Infrastructure categories are not invented. They are named. They emerge when a problem becomes so pervasive that ad-hoc solutions can no longer contain it, and someone defines the problem precisely enough that a standard solution becomes possible.

TCP/IP did not invent networking. Networks existed. But TCP/IP named the problem — "how do heterogeneous networks interoperate?" — and defined a standard protocol stack that every network could implement. Today, TCP/IP is invisible infrastructure that everything depends on.

TLS did not invent encryption. Encryption existed. But TLS named the problem — "how do two parties establish a secure channel over an untrusted network?" — and defined a standard handshake protocol. Today, HTTPS is so ubiquitous that browsers warn users when it is absent.

OAuth did not invent authorization. Authorization existed. But OAuth named the problem — "how does a user grant an application limited access to their resources without sharing credentials?" — and defined a standard delegation flow. Today, "Sign in with Google" is the default.

JWT did not invent identity claims. Identity claims existed. But JWT named the format — "how do we represent identity as a portable, self-verifying, signed token?" — and defined a standard that every API now speaks.

In each case, the pattern is the same: a pervasive problem exists, individual teams build custom solutions, someone names the problem and defines a standard, and the standard becomes invisible infrastructure. The naming is the breakthrough. Before the name exists, the problem is fragmented. After the name exists, the category crystallizes.

基礎設施類別不是被發明的。它們是被命名的。它們在一個問題變得如此普遍以至於臨時解決方案無法再應對時出現，而某人足夠精確地定義了問題，使標準解決方案成為可能。

TCP/IP 沒有發明網路。網路已經存在。但 TCP/IP 命名了問題——「異構網路如何互操作？」——並定義了每個網路都可以實作的標準協議堆疊。今天，TCP/IP 是一切都依賴的隱形基礎設施。

TLS 沒有發明加密。加密已經存在。但 TLS 命名了問題——「兩方如何在不受信任的網路上建立安全通道？」——並定義了標準握手協議。今天，HTTPS 如此普遍，以至於瀏覽器在缺少它時會警告用戶。

OAuth 沒有發明授權。授權已經存在。但 OAuth 命名了問題——「用戶如何在不分享憑證的情況下授予應用程式有限的資源存取權？」——並定義了標準委託流程。今天，「使用 Google 登入」是預設選項。

JWT 沒有發明身份聲明。身份聲明已經存在。但 JWT 命名了格式——「如何將身份表示為可攜帶、自驗證、已簽名的令牌？」——並定義了每個 API 現在都使用的標準。

在每種情況下，模式是相同的：普遍的問題存在、個別團隊建構自定解決方案、某人命名問題並定義標準、標準成為隱形基礎設施。命名就是突破。在名稱存在之前，問題是碎片化的。在名稱存在之後，類別結晶。

7.2 — Computational Trust Is the Next Category

7.2 — 計算信任是下一個類別

The problem is here. Autonomous systems are proliferating faster than the trust infrastructure to support them. Every major organization deploying AI agents is building custom solutions for decision auditability — custom logging, custom attestation, custom compliance mapping. These solutions are fragmented, incompatible, and insufficient.

The regulatory demand is here. Every major regulatory body — EU, US, UK, Singapore, Japan — is independently converging on the same requirement: AI decisions must be auditable and verifiable. But no regulatory framework specifies how, because the infrastructure to do so does not exist.

The name is here. Computational Trust — the ability to independently verify that a computational process occurred correctly, fairly, and without manipulation. This is not security (protecting the perimeter), not privacy (protecting data), not compliance (following rules). It is a distinct infrastructure layer that sits alongside all three.

OpenRNG is naming this category and building its first implementation. The Trust Stack defines the architecture. The Trust Lifecycle defines the operations. The VEO-1 specification defines the object format. The patent family protects the intellectual property. The SDK and open standard enable adoption.

問題已經在這裡。自主系統的擴散速度超過了支撐它們的信任基礎設施。每個部署 AI 代理的主要組織都在為決策可稽核性建構自定解決方案——自定日誌、自定證明、自定合規對應。這些解決方案是碎片化的、不相容的、不充分的。

監管需求已經在這裡。每個主要監管機構——歐盟、美國、英國、新加坡、日本——都獨立地趨同於相同的要求：AI 決策必須是可稽核和可驗證的。但沒有任何監管框架指定如何做到，因為做到這一點的基礎設施不存在。

名稱已經在這裡。計算信任——獨立驗證計算過程正確、公平且未被操縱地發生的能力。這不是安全（保護邊界）、不是隱私（保護資料）、不是合規（遵循規則）。它是一個獨特的基礎設施層，與這三者並列存在。

OpenRNG 正在命名這個類別並建構其第一個實作。信任堆疊定義架構。信任生命週期定義操作。VEO-1 規範定義物件格式。專利家族保護智慧財產權。SDK 和開放標準實現採用。
