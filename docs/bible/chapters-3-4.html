3.1 — Design Principles

3.1 — 設計原則

Before describing the architecture, we must establish the principles that constrain it. These principles are not aspirational — they are engineering requirements. Any architectural decision that violates a principle is a defect.

Principle 1: Trust Must Be Portable. A trust object must be verifiable anywhere — across runtimes, across organizations, across jurisdictions. A VEO generated on AWS must be verifiable on Azure, on-premises, or on a developer's laptop. Trust cannot be locked to a platform, a chain, or a provider. This is the portability property that makes JWT useful: a JWT issued by Auth0 can be verified by any service with the public key. Trust objects must have the same property.

Principle 2: Trust Must Be Independently Verifiable. Verification must never require contacting the issuer. No callbacks, no phone-home, no "check with our API." A trust object carries its own proof. Given the object and the provider's public key, any party can verify it — offline, air-gapped, years later. This is what distinguishes infrastructure from a service: infrastructure works even when the provider is unreachable.

Principle 3: Trust Must Be Composable. A trust object from one computation can be an input to another computation, creating trust chains. An AI agent that receives a VEO can use it to make a decision, produce a new VEO for that decision, and pass it to the next agent. Each link in the chain is independently verifiable. Composability is what makes trust scale — without it, every interaction requires a fresh root of trust.

Principle 4: Trust Must Be Measurable. Trust is not binary. A trust object backed by three independent entropy sources and blockchain anchoring is more trustworthy than one backed by a single PRNG. The Entropy Confidence Score (ECS) quantifies this difference across six dimensions, producing a score from 0 to 1000 with defined grade boundaries. Measurability enables policy: "this application requires AAA-grade trust" becomes an enforceable constraint.

Principle 5: Trust Must Be Standard. A proprietary trust format is not infrastructure — it is a product. For Computational Trust to become a category, the object format, scoring system, signing protocol, and verification process must be open, documented, and implementable by anyone. The VEO-1 specification, ECS v1, and the canonical signing protocol are designed as open standards from the outset.

在描述架構之前，我們必須建立約束架構的原則。這些原則不是願景——而是工程需求。任何違反原則的架構決策都是缺陷。

原則一：信任必須可攜帶。信任物件必須在任何地方都可驗證——跨執行環境、跨組織、跨司法管轄區。在 AWS 上生成的 VEO 必須能在 Azure、本地環境或開發者的筆記型電腦上驗證。信任不能被鎖定在特定平台、鏈或供應商。這就是讓 JWT 有用的可攜帶性：Auth0 發行的 JWT 可以被任何擁有公鑰的服務驗證。信任物件必須具有相同的性質。

原則二：信任必須可獨立驗證。驗證永遠不需要聯繫發行者。沒有回呼、沒有回報、沒有「請向我們的 API 確認」。信任物件自帶證明。給定物件和供應商的公鑰，任何一方都可以驗證——離線、氣隙隔離、甚至多年之後。這就是基礎設施與服務的區別：基礎設施在供應商無法連線時仍然運作。

原則三：信任必須可組合。一個計算的信任物件可以成為另一個計算的輸入，形成信任鏈。收到 VEO 的 AI 代理可以用它做出決策，為該決策產生新的 VEO，並傳遞給下一個代理。鏈中的每個環節都可獨立驗證。可組合性讓信任得以擴展——沒有它，每次互動都需要全新的信任根。

原則四：信任必須可量測。信任不是二元的。由三個獨立熵源和區塊鏈錨定支持的信任物件，比僅由單一 PRNG 支持的信任物件更值得信賴。熵信心分數（ECS）跨六個維度量化這個差異，產生 0 到 1000 的分數，並定義分級邊界。可量測性使策略成為可能：「此應用程式需要 AAA 級信任」成為可執行的約束。

原則五：信任必須是標準。專有的信任格式不是基礎設施——而是產品。要讓計算信任成為一個類別，物件格式、評分系統、簽名協議和驗證流程必須是開放的、有文件記錄的、任何人都可以實作的。VEO-1 規範、ECS v1 和規範簽名協議從一開始就被設計為開放標準。

3.3 — Layer 1: Trust Source (OpenRNG Core)

3.3 — 第一層：信任來源（OpenRNG 核心）

The Trust Source layer is the foundation of the entire stack. Its responsibility is singular: produce high-quality, multi-source entropy as the raw material of trust. Everything above this layer — authorization, evidence, coordination, governance — depends on the quality and integrity of what this layer produces.

The layer operates through a four-stage pipeline:

Stage 1: Source Collection. The engine queries multiple independent entropy sources in parallel. Current production sources include drand (a distributed randomness beacon operated by the League of Entropy, including Cloudflare, Protocol Labs, and academic institutions), Bitcoin block hashes (proof-of-work entropy derived from the computational effort of the entire Bitcoin mining network), and Polygon block hashes (proof-of-stake entropy from validator consensus). The architecture is source-agnostic: future sources include Quantum Random Number Generators (QRNG), additional randomness beacons, and federated source networks.

Stage 2: Source Verification. Each source response is independently verified before inclusion. drand beacon signatures are checked against the published League of Entropy public key. Blockchain hashes are verified against on-chain state. Sources that fail verification are flagged but do not halt generation — they reduce the Entropy Confidence Score, providing transparent degradation rather than opaque failure.

Stage 3: Entropy Aggregation. Verified source entropies are combined using a deterministic aggregation function. The current implementation uses concatenation followed by SHA-256 hashing, producing a single entropy value that incorporates contributions from all verified sources. The aggregation method is recorded in the output for auditability.

Stage 4: Quality Scoring. The Entropy Confidence Score (ECS) is computed across six independently measured dimensions: freshness (time since generation), diversity (number of unique source identifiers), independence (number of unique source types), manipulation resistance (difficulty of controlling the output), verification success (proportion of sources that passed verification), and availability (source accessibility at generation time). The composite score ranges from 0 to 1000 with defined grade boundaries from AAA (≥900) to LOW (<500).

The Trust Source layer also implements batch generation via Merkle trees — a single VDF (Verifiable Delay Function) seed can deterministically generate up to 65,536 entropy values organized in a Merkle tree, enabling sub-2ms serving latency while maintaining cryptographic proof for each individual value. This batch architecture is critical for production-scale deployment where latency budgets are measured in milliseconds.

信任來源層是整個堆疊的基礎。它的職責是單一的：產生高品質、多來源的熵作為信任的原始材料。此層之上的一切——授權、證據、協調、治理——都取決於此層產出的品質和完整性。

此層透過四階段管線運作：

階段一：來源收集。引擎平行查詢多個獨立熵源。當前生產環境來源包括 drand（由 Cloudflare、Protocol Labs 和學術機構等組成的 League of Entropy 運營的分散式隨機信標）、比特幣區塊雜湊（來自整個比特幣挖礦網路計算努力的工作量證明熵），以及 Polygon 區塊雜湊（來自驗證者共識的權益證明熵）。架構是來源無關的：未來來源包括量子隨機數生成器（QRNG）、額外的隨機信標和聯邦來源網路。

階段二：來源驗證。每個來源回應在納入前都會獨立驗證。drand 信標簽名會對照已公開的 League of Entropy 公鑰進行檢查。區塊鏈雜湊會對照鏈上狀態進行驗證。未通過驗證的來源會被標記，但不會中斷生成——它們會降低熵信心分數，提供透明的降級而非不透明的失敗。

階段三：熵聚合。經驗證的來源熵使用確定性聚合函數組合。當前實作使用串聯後進行 SHA-256 雜湊，產生包含所有已驗證來源貢獻的單一熵值。聚合方法記錄在輸出中以供稽核。

階段四：品質評分。熵信心分數（ECS）跨六個獨立量測的維度計算：新鮮度（生成後經過的時間）、多樣性（唯一來源識別碼數量）、獨立性（唯一來源類型數量）、抗操縱性（控制輸出的難度）、驗證成功率（通過驗證的來源比例），以及可用性（生成時的來源可存取性）。綜合分數範圍從 0 到 1000，定義的等級邊界從 AAA（≥900）到 LOW（<500）。

信任來源層還透過 Merkle 樹實作批次生成——單一 VDF（可驗證延遲函數）種子可以確定性地生成多達 65,536 個熵值，組織在 Merkle 樹中，實現低於 2 毫秒的服務延遲，同時為每個個別值維持密碼學證明。這種批次架構對於延遲預算以毫秒計算的生產規模部署至關重要。

3.4 — Layer 2: Trust Authorization (Lockbox)

3.4 — 第二層：信任授權（Lockbox）

The Trust Authorization layer — codenamed Lockbox — controls who may consume trust primitives and under what conditions. This is not traditional access control. Traditional access control (OAuth, RBAC, API keys) answers a single question: "Is this identity allowed to access this resource?" Trust Authorization answers a different question: "Is this consumer allowed to use this trust primitive, and can we prove they used it correctly?"

The distinction is attestation. OAuth grants access and forgets. Lockbox grants access and records cryptographic proof of correct usage. Every consumption event is attested — creating an irrefutable record that a specific consumer used a specific trust primitive at a specific time under specific policy constraints.

Lockbox enforces three categories of policy:

Consumption Policies: Rate limiting (maximum entropy requests per time window), quality floors (minimum ECS grade required for the consumer's use case), source constraints (which entropy source types are permitted or excluded), and anchoring requirements (whether blockchain anchoring is mandatory).

Delegation Policies: Whether a consumer can delegate their trust authorization to sub-agents, under what conditions, and with what constraints. This is critical for multi-agent AI systems where a primary agent spawns child agents — the child must inherit authorization within defined boundaries.

Compliance Policies: Regulatory-context-specific constraints that map to frameworks like GLI-19 (gaming), EU AI Act (AI systems), or HIPAA (healthcare). These policies ensure that trust primitives consumed in regulated contexts meet the appropriate requirements automatically, without requiring the consumer to understand the regulatory details.

信任授權層——代號 Lockbox——控制誰可以消費信任原語，以及在什麼條件下消費。這不是傳統的存取控制。傳統存取控制（OAuth、RBAC、API 金鑰）回答一個問題：「此身份是否被允許存取此資源？」信任授權回答不同的問題：「此消費者是否被允許使用此信任原語，我們能否證明他們正確使用了它？」

區別在於證明。OAuth 授予存取權然後遺忘。Lockbox 授予存取權並記錄正確使用的密碼學證明。每個消費事件都被證明——建立不可否認的記錄：特定消費者在特定時間、特定策略約束下使用了特定信任原語。

Lockbox 執行三類策略：

消費策略：速率限制（每時間窗口最大熵請求數）、品質下限（消費者使用案例所需的最低 ECS 等級）、來源約束（允許或排除哪些熵源類型），以及錨定需求（是否強制要求區塊鏈錨定）。

委託策略：消費者是否可以將信任授權委託給子代理、在什麼條件下、以及有什麼約束。這對多代理 AI 系統至關重要——當主要代理產生子代理時，子代理必須在定義的邊界內繼承授權。

合規策略：對應到 GLI-19（遊戲）、EU AI Act（AI 系統）或 HIPAA（醫療保健）等框架的法規環境特定約束。這些策略確保在受監管環境中消費的信任原語自動滿足適當的要求，無需消費者理解法規細節。

3.5 — Layer 3: Trust Evidence (VEO)

3.5 — 第三層：信任證據（VEO）

The Trust Evidence layer is the heart of the stack — the layer that transforms raw entropy into a standardized, self-describing, independently verifiable digital object. This layer produces the Verifiable Entropy Object (VEO-1), the core primitive of Computational Trust Infrastructure.

A VEO is not a random number with metadata. It is a proof-carrying object — a cryptographic container that enables any party, anywhere, at any time, to independently verify the integrity, provenance, quality, and authenticity of the entropy it carries, without trusting the issuer or contacting any external service.

The VEO-1 specification defines four object classes: VEO-1A (single-source raw entropy), VEO-1B (multi-source composite entropy), VEO-1C (blockchain-anchored entropy with Merkle proof or transaction hash), and VEO-1D (reserved for Verifiable Decision Objects — entropy objects extended with decision context). Each class represents a different level of trust evidence.

Every VEO carries five layers of information:

Payload: The entropy value itself, plus its SHA-256 hash. Any verifier can recompute the hash from the payload to confirm integrity.

Provenance: An array of source records, each containing source identifier, source type, reference (e.g., "drand-round-29830144"), timestamp, and per-source entropy hash. This creates a complete audit trail of where every bit of entropy originated.

Confidence: The Entropy Confidence Score (ECS) — a composite score (0–1000) computed across six weighted dimensions, with a letter grade from AAA to LOW. The score is deterministically reproducible: given the same source records, any party can independently recompute the same ECS.

Proof: A secp256k1 EIP-191 provider signature over the canonicalized VEO payload. Canonicalization uses deep key sorting and deterministic serialization to ensure that the same VEO always produces the same signing payload, regardless of JSON key ordering. The signature can be verified against the provider's public key without any network access.

Anchor (optional): A blockchain anchor — either a direct transaction hash or a Merkle proof linking the VEO to an on-chain Merkle root. The Merkle batch system enables a single on-chain transaction to anchor up to 65,536 VEOs, providing temporal proof (the VEO existed at or before the block timestamp) and immutability (the anchor cannot be altered after confirmation).

The VEO-1 verification framework defines five levels:

信任證據層是堆疊的核心——將原始熵轉化為標準化、自描述、可獨立驗證的數位物件的層。此層產生可驗證熵物件（VEO-1），計算信任基礎設施的核心原語。

VEO 不是附帶元數據的隨機數。它是一個攜帶證明的物件——一個密碼學容器，使任何一方、在任何地方、在任何時間，都能獨立驗證其攜帶的熵的完整性、出處、品質和真實性，無需信任發行者或聯繫任何外部服務。

VEO-1 規範定義了四個物件類別：VEO-1A（單來源原始熵）、VEO-1B（多來源複合熵）、VEO-1C（具 Merkle 證明或交易雜湊的區塊鏈錨定熵），以及 VEO-1D（保留給可驗證決策物件——擴展了決策上下文的熵物件）。每個類別代表不同層級的信任證據。

每個 VEO 攜帶五層資訊：

載荷：熵值本身，加上其 SHA-256 雜湊。任何驗證者都可以從載荷重新計算雜湊以確認完整性。

出處：一組來源記錄，每條包含來源識別碼、來源類型、參考（如「drand-round-29830144」）、時間戳和每來源熵雜湊。這為每一位元熵的起源建立了完整的稽核軌跡。

信心：熵信心分數（ECS）——跨六個加權維度計算的綜合分數（0-1000），附字母等級從 AAA 到 LOW。分數是確定性可重現的：給定相同的來源記錄，任何一方都可以獨立重新計算相同的 ECS。

證明：對規範化 VEO 載荷的 secp256k1 EIP-191 供應商簽名。規範化使用深層鍵排序和確定性序列化，確保相同的 VEO 總是產生相同的簽名載荷，無論 JSON 鍵的排序如何。簽名可以在沒有任何網路存取的情況下對照供應商的公鑰進行驗證。

錨定（可選）：區塊鏈錨定——直接交易雜湊或將 VEO 連結到鏈上 Merkle 根的 Merkle 證明。Merkle 批次系統使單一鏈上交易能夠錨定多達 65,536 個 VEO，提供時間證明（VEO 在區塊時間戳之前或之時存在）和不可變性（錨定在確認後無法更改）。

3.6 — Layer 4: Trust Coordination

3.6 — 第四層：信任協調

Layers 1 through 3 solve the trust problem for individual computations: a single agent, a single decision, a single verifiable object. But modern AI systems are not single-agent. They are swarms — dozens or hundreds of autonomous agents collaborating, delegating, and handing off tasks. When Agent A assigns a task to Agent B based on verifiable entropy, Agent B needs to trust that the assignment was fair — and Agent C (the auditor) needs to verify the entire chain.

The Trust Coordination layer solves the horizontal trust propagation problem: how does trust flow between autonomous agents and across system boundaries?

Trust Chain Construction. When a VEO is used to make a decision that spawns child computations, the child VEOs reference the parent via the lineage field. This creates a directed acyclic graph (DAG) of trust — every decision can be traced back to its root entropy. A task allocation VEO spawns sub-task VEOs, which spawn tool-selection VEOs, forming a complete, auditable decision tree.

Multi-Agent Trust Consensus. When multiple agents must agree on a shared random value — for example, jointly selecting a leader, allocating a contested resource, or establishing turn order — Trust Coordination provides a protocol for combining individual VEOs into a consensus VEO. Each participant contributes entropy; the consensus VEO references all contributions; any party can verify that the consensus was computed correctly from the individual inputs.

Cross-System Trust Handoff. When trust needs to cross organizational boundaries — an enterprise AI system delegating to a third-party agent service — Trust Coordination defines the protocol for packaging trust assertions, transmitting them across trust boundaries, and verifying them on the receiving side. The handoff protocol ensures that trust does not require mutual trust between organizations — only trust in the cryptographic proofs.

Trust Propagation Rules. Three invariants govern trust propagation: (1) Trust does not amplify — a child VEO's ECS cannot exceed its parent's; (2) Trust degrades transparently — if a child uses fewer sources, the degradation is recorded, not hidden; (3) Trust is traceable — every VEO in a chain can be independently verified without trusting intermediate nodes.

It is critical to understand what Trust Coordination is not. It is not an orchestrator. It does not control agents or decide what they do. It is a protocol for trust transmission — analogous to TCP for data transmission. TCP does not decide what data to send; it ensures data arrives correctly. Trust Coordination does not decide what agents do; it ensures trust propagates correctly.

第一到三層解決的是個別計算的信任問題：單一代理、單一決策、單一可驗證物件。但現代 AI 系統不是單一代理的。它們是群體——數十或數百個自主代理協作、委託和交接任務。當代理 A 基於可驗證熵將任務分配給代理 B 時，代理 B 需要信任分配是公平的——而代理 C（稽核員）需要驗證整條鏈。

信任協調層解決的是水平信任傳播問題：信任如何在自主代理之間和跨系統邊界流動？

信任鏈建構。當 VEO 被用於做出產生子計算的決策時，子 VEO 透過血統欄位參考父代。這建立了信任的有向無環圖（DAG）——每個決策都可以追溯到其根熵。任務分配 VEO 產生子任務 VEO，再產生工具選擇 VEO，形成完整、可稽核的決策樹。

多代理信任共識。當多個代理必須就共享隨機值達成一致——例如共同選舉領導者、分配爭議資源或確定順序——信任協調提供將個別 VEO 組合為共識 VEO 的協議。每個參與者貢獻熵；共識 VEO 參考所有貢獻；任何一方都可以驗證共識是否從個別輸入正確計算。

跨系統信任交接。當信任需要跨越組織邊界——企業 AI 系統委託給第三方代理服務——信任協調定義了封裝信任斷言、跨越信任邊界傳輸、並在接收端驗證的協議。交接協議確保信任不需要組織間的相互信任——只需要信任密碼學證明。

信任傳播規則。三個不變量治理信任傳播：(1) 信任不會放大——子 VEO 的 ECS 不能超過其父代；(2) 信任透明降級——如果子代使用更少的來源，降級被記錄而非隱藏；(3) 信任可追溯——鏈中的每個 VEO 都可以在不信任中間節點的情況下獨立驗證。

理解信任協調不是什麼至關重要。它不是編排器。它不控制代理或決定它們做什麼。它是信任傳輸協議——類似於 TCP 之於資料傳輸。TCP 不決定發送什麼資料；它確保資料正確到達。信任協調不決定代理做什麼；它確保信任正確傳播。

3.7 — Layer 5: Trust Runtime

3.7 — 第五層：信任運行時

The Trust Runtime layer provides governance, compliance, and continuous verification for the entire trust infrastructure. While Layer 2 (Trust Authorization) enforces policy at the point of consumption — a per-request gate — the Trust Runtime enforces policy across the entire lifecycle, retroactively, continuously, and against evolving requirements.

Runtime Policy Enforcement. The Trust Runtime monitors and enforces system-wide policies that cannot be evaluated at the per-request level: maximum trust chain depth (e.g., no VEO lineage deeper than five levels), source rotation requirements (e.g., no single source may dominate more than 70% of requests over any 24-hour window), anchoring frequency (e.g., all VEOs above a certain value threshold must be anchored within 60 seconds), and temporal consistency (e.g., no VEO timestamps may be out of sequence).

Compliance Attestation. The Trust Runtime generates structured compliance attestations that map trust infrastructure operations to regulatory framework requirements. For the EU AI Act, the runtime demonstrates that AI decisions backed by VEOs satisfy Article 14 (Human Oversight) transparency requirements. For NIST AI RMF, the trust lifecycle maps directly to the GOVERN function. For GLI-19 (Gaming Laboratories International), VEO verification satisfies RNG certification requirements. These attestations are machine-readable and designed for integration with enterprise compliance systems.

Trust Health Monitoring. The runtime continuously monitors trust infrastructure health metrics: system-wide average ECS over time (detecting source degradation), source availability and diversity trends, verification success rates, mean time between anchoring, trust chain depth distribution, and consumption patterns. Anomalies trigger alerts — for example, a sustained decline in average ECS may indicate that an entropy source is degrading, requiring investigation before it affects downstream trust quality.

The boundary between Layer 4 (Trust Coordination) and Layer 5 (Trust Runtime) is precise and intentional: Coordination is horizontal — it governs how trust flows between peers. Runtime is vertical — it governs how trust is enforced, monitored, and attested from top to bottom. Coordination is a protocol. Runtime is a governance engine.

信任運行時層為整個信任基礎設施提供治理、合規和持續驗證。第二層（信任授權）在消費點執行策略——逐請求的閘門——而信任運行時跨越整個生命週期執行策略，具有追溯性、持續性，並適應不斷演變的要求。

運行時策略執行。信任運行時監控並執行無法在逐請求層級評估的系統範圍策略：最大信任鏈深度（如 VEO 血統不超過五層）、來源輪換要求（如任何 24 小時窗口內單一來源不得主導超過 70% 的請求）、錨定頻率（如超過特定價值閾值的所有 VEO 必須在 60 秒內錨定），以及時間一致性（如 VEO 時間戳不得亂序）。

合規證明。信任運行時生成結構化的合規證明，將信任基礎設施操作對應到法規框架要求。對於 EU AI Act，運行時證明由 VEO 支持的 AI 決策滿足第 14 條（人類監督）透明度要求。對於 NIST AI RMF，信任生命週期直接對應到 GOVERN 功能。對於 GLI-19（遊戲實驗室國際），VEO 驗證滿足 RNG 認證要求。這些證明是機器可讀的，設計用於與企業合規系統整合。

信任健康監控。運行時持續監控信任基礎設施健康指標：系統範圍的平均 ECS 隨時間變化（偵測來源降級）、來源可用性和多樣性趨勢、驗證成功率、平均錨定間隔時間、信任鏈深度分佈，以及消費模式。異常觸發警報——例如，平均 ECS 的持續下降可能表明某個熵源正在降級，需要在影響下游信任品質之前進行調查。

第四層（信任協調）和第五層（信任運行時）之間的邊界是精確且有意的：協調是水平的——治理信任如何在對等方之間流動。運行時是垂直的——治理信任如何從上到下被執行、監控和證明。協調是協議。運行時是治理引擎。

3.8 — Cross-Cutting Concerns

3.8 — 跨層關注事項

Versioning and Backward Compatibility. The Trust Stack follows a strict versioning policy. The VEO-1 specification and ECS v1 scoring algorithm are frozen — required fields, dimension weights, and grade boundaries cannot change without a protocol version bump. New optional fields may be added in minor versions. This stability is critical for trust: a VEO issued today must be verifiable ten years from now using the same verification logic.

Performance. Trust infrastructure must not impose unacceptable latency. The Trust Source layer achieves sub-2ms serving through Merkle batch pre-generation — entropy values are generated ahead of demand and served from a pre-computed pool. The Evidence layer adds approximately 1ms for signing. Blockchain anchoring is asynchronous and does not block the critical path. The total overhead for a Standard-profile interaction (Require → Generate → Authorize → Execute → Evidence) is under 5ms — negligible for any production use case.

Failure Modes. The Trust Stack is designed for transparent degradation. When an entropy source is unavailable, the system continues with available sources and reduces the ECS score accordingly. When blockchain anchoring fails, the VEO is issued at a lower verification level (cryptographically_verified instead of anchored_verified) with the failure reason recorded. There are no silent failures — every degradation is visible in the VEO's confidence and proof fields.

Extensibility. Each layer is designed for extension without modification. New entropy sources can be added to Layer 1 by implementing a source adapter. New authorization policies can be added to Layer 2 through the policy engine. New verification types can be added to Layer 3 by extending the proof package. The five-layer structure itself is stable; what changes is the implementation within each layer.

版本控制和向後相容性。信任堆疊遵循嚴格的版本控制策略。VEO-1 規範和 ECS v1 評分演算法已凍結——必填欄位、維度權重和等級邊界在沒有協議版本升級的情況下不能更改。新的可選欄位可以在次要版本中添加。這種穩定性對信任至關重要：今天發行的 VEO 必須在十年後使用相同的驗證邏輯仍可驗證。

性能。信任基礎設施不得施加不可接受的延遲。信任來源層透過 Merkle 批次預生成實現低於 2ms 的服務——熵值在需求之前生成，並從預計算池中提供。證據層為簽名增加約 1ms。區塊鏈錨定是非同步的，不阻塞關鍵路徑。標準配置互動（需求 → 生成 → 授權 → 執行 → 證據）的總開銷低於 5ms——對任何生產使用案例來說可以忽略不計。

故障模式。信任堆疊設計為透明降級。當熵源不可用時，系統繼續使用可用來源並相應降低 ECS 分數。當區塊鏈錨定失敗時，VEO 以較低的驗證層級發行（cryptographically_verified 而非 anchored_verified），並記錄失敗原因。沒有靜默失敗——每次降級都在 VEO 的信心和證明欄位中可見。

可擴展性。每一層都設計為可擴展而無需修改。新的熵源可以透過實作來源適配器添加到第一層。新的授權策略可以透過策略引擎添加到第二層。新的驗證類型可以透過擴展證明包添加到第三層。五層結構本身是穩定的；變化的是每層內的實作。

4.1 — Why a Lifecycle?

4.1 — 為什麼需要生命週期？

Every infrastructure category has a lifecycle — a canonical sequence of operations that defines how the infrastructure is used, not just what it is.

TLS has a lifecycle: handshake → key exchange → encrypted session → termination. OAuth has a lifecycle: authorization request → user consent → token issuance → resource access → token refresh → revocation. DNS has a lifecycle: query → recursive resolution → caching → TTL expiry → re-resolution.

These lifecycles are what make infrastructure operational. Without them, a technology is just a specification. With them, it becomes something engineers build systems around.

Computational Trust Infrastructure is no different. The Trust Stack (Chapter 3) defines the architecture — the layers, their responsibilities, their interfaces. The Trust Lifecycle defines how those layers operate in sequence to produce, propagate, and verify trust.

If the Trust Stack is the anatomy of the system, the Trust Lifecycle is its physiology.

每個基礎設施類別都有一個生命週期——定義基礎設施如何使用（而不僅僅是它是什麼）的規範操作序列。

TLS 有生命週期：握手 → 金鑰交換 → 加密會話 → 終止。OAuth 有生命週期：授權請求 → 使用者同意 → 令牌發行 → 資源存取 → 令牌刷新 → 撤銷。DNS 有生命週期：查詢 → 遞迴解析 → 快取 → TTL 過期 → 重新解析。

這些生命週期使基礎設施具有可操作性。沒有它們，技術只是規範。有了它們，它就成為工程師建構系統的依據。

計算信任基礎設施也不例外。信任堆疊（第三章）定義了架構——層級、其職責、其介面。信任生命週期定義了這些層如何依序運作來產生、傳播和驗證信任。

如果信任堆疊是系統的解剖學，信任生命週期就是其生理學。

4.2 — The Eight Stages

4.2 — 八個階段

The Trust Lifecycle consists of eight stages. Not every use case requires all eight. The lifecycle is designed to be truncatable — simpler use cases use fewer stages, and the system remains valid at every truncation point.

每個階段轉換一個信任狀態。每次轉換都產生一個可驗證的成品。生命週期不是管線——它是在每個階段都有定義不變量的狀態機。

Stage 1: REQUIRE — 需求

What trust does this computation need?

A system — an AI agent, a game server, a clinical trial platform — encounters a decision point that requires non-deterministic input. Before generating anything, the system must articulate what kind of trust it needs: quality threshold (minimum ECS grade), source constraints (which entropy sources are acceptable), verification level (structural, cryptographic, or anchored), latency budget, anchoring requirements, and regulatory context.

The output is a Trust Requirement Object (TRO) — a structured declaration of what the consumer needs. The TRO is immutable once issued; if requirements change, a new TRO must be created, ensuring audit trails can always trace back to the original requirement.

此計算需要什麼信任？

一個系統——AI 代理、遊戲伺服器、臨床試驗平台——遇到需要非確定性輸入的決策點。在生成任何東西之前，系統必須清楚表達它需要什麼類型的信任：品質閾值（最低 ECS 等級）、來源約束（可接受哪些熵源）、驗證層級（結構性、密碼學或錨定）、延遲預算、錨定要求和法規背景。

輸出是信任需求物件（TRO）——消費者需求的結構化聲明。TRO 一旦發出即不可變；如果需求變更，必須建立新的 TRO，確保稽核軌跡始終可以追溯到原始需求。

Stage 2: GENERATE — 生成

Where does the raw trust material come from?

The Trust Source layer (Layer 1) produces raw entropy through a four-stage pipeline: source collection (querying multiple independent entropy sources in parallel), source verification (independently verifying each response — drand signatures, blockchain hashes), entropy aggregation (deterministic combination via concatenation + SHA-256), and quality scoring (computing the six-dimension ECS). The generation process is deterministic given the same source inputs — any auditor with access to the source records can re-derive the aggregated entropy and verify the ECS score.

原始信任材料從哪裡來？

信任來源層（第一層）透過四階段管線產生原始熵：來源收集（平行查詢多個獨立熵源）、來源驗證（獨立驗證每個回應——drand 簽名、區塊鏈雜湊）、熵聚合（透過串聯 + SHA-256 的確定性組合），以及品質評分（計算六維度 ECS）。生成過程在給定相同來源輸入時是確定性的——任何有權存取來源記錄的稽核員都可以重新導出聚合熵並驗證 ECS 分數。

Stage 3: AUTHORIZE — 授權

Is this consumer allowed to use this trust primitive, under these conditions?

The Trust Authorization layer (Layer 2 — Lockbox) enforces policy on consumption. Unlike traditional access control, Trust Authorization produces attestation — cryptographic proof that a specific consumer used a specific trust primitive at a specific time under specific policy constraints. Every consumption event is attested. There is no "anonymous" entropy consumption in a trust infrastructure.

此消費者是否被允許在這些條件下使用此信任原語？

信任授權層（第二層——Lockbox）對消費執行策略。與傳統存取控制不同，信任授權產生證明——特定消費者在特定時間、特定策略約束下使用了特定信任原語的密碼學證明。每個消費事件都被證明。在信任基礎設施中沒有「匿名」熵消費。

Stage 4: EXECUTE — 執行

What computation was performed using the trust primitive?

The consumer uses the authorized trust primitive in their computation — an AI agent selects a tool, a casino determines an outcome, a trial platform randomizes patients. This stage is intentionally outside the Trust Stack's direct control. The stack provides the primitive; the consumer provides the computation. This separation is critical for adoption: consumers should not need to restructure their applications to use verifiable entropy.

使用信任原語執行了什麼計算？

消費者在其計算中使用授權的信任原語——AI 代理選擇工具、賭場確定結果、試驗平台隨機分配患者。此階段有意置於信任堆疊的直接控制之外。堆疊提供原語；消費者提供計算。這種分離對採用至關重要：消費者不應需要重組其應用程式來使用可驗證熵。

Stage 5: EVIDENCE — 證據

Can we package what happened into a verifiable, portable proof?

The Trust Evidence layer (Layer 3) transforms the computation into a VEO-1 object through four operations: object construction (assembling payload, sources, ECS), canonical signing (deterministic serialization + secp256k1 EIP-191 signature), blockchain anchoring (optional Merkle batch on Polygon), and verification level assignment. The resulting VEO is self-verifying: given only the object and the provider's public key, any party can verify integrity, authenticity, quality, and temporal anchoring — offline, air-gapped, years later.

我們能否將發生的事情封裝為可驗證、可攜帶的證明？

信任證據層（第三層）透過四個操作將計算轉化為 VEO-1 物件：物件建構（組裝載荷、來源、ECS）、規範簽名（確定性序列化 + secp256k1 EIP-191 簽名）、區塊鏈錨定（可選的 Polygon Merkle 批次），以及驗證層級指定。產出的 VEO 是自驗證的：僅憑物件和供應商的公鑰，任何一方都可以驗證完整性、真實性、品質和時間錨定——離線、氣隙隔離、多年之後。

Stage 6: COORDINATE — 協調

How does trust flow between multiple agents and systems?

In multi-agent AI systems, trust must propagate across autonomous agents. Trust Coordination enables trust chain construction (child VEOs reference parents via lineage, forming a DAG), multi-agent trust consensus (combining individual VEOs into a jointly verifiable consensus), and cross-system trust handoff (packaging trust assertions for transmission across organizational boundaries). Three invariants govern propagation: trust does not amplify, trust degrades transparently, and trust is always traceable.

信任如何在多個代理和系統之間流動？

在多代理 AI 系統中，信任必須跨自主代理傳播。信任協調實現信任鏈建構（子 VEO 透過血統參考父代，形成 DAG）、多代理信任共識（將個別 VEO 組合為可聯合驗證的共識），以及跨系統信任交接（封裝信任斷言以跨組織邊界傳輸）。三個不變量治理傳播：信任不會放大、信任透明降級、信任始終可追溯。

Stage 7: GOVERN — 治理

Does this trust chain comply with policy and regulation?

The Trust Runtime (Layer 5) provides continuous governance: runtime policy enforcement (trust chain depth limits, source rotation requirements, anchoring frequency), compliance attestation (mapping to EU AI Act, NIST AI RMF, GLI-19, ISO 42001), anomaly detection (ECS drift, source concentration, consumption pattern analysis), and trust health monitoring. Authorization (Stage 3) is a per-request gate; Governance is a continuous monitor across the entire lifecycle.

此信任鏈是否符合策略和法規？

信任運行時（第五層）提供持續治理：運行時策略執行（信任鏈深度限制、來源輪換要求、錨定頻率）、合規證明（對應 EU AI Act、NIST AI RMF、GLI-19、ISO 42001）、異常偵測（ECS 漂移、來源集中度、消費模式分析），以及信任健康監控。授權（階段三）是逐請求的閘門；治理是跨越整個生命週期的持續監控器。

Stage 8: AUDIT — 稽核

Can an independent third party verify everything that happened?

The reason the entire lifecycle exists. An auditor can verify any individual VEO (recompute hash, verify signature, reproduce ECS, validate Merkle proof), traverse trust chains (follow lineage from any VEO to its root), verify policy compliance (check every VEO against its TRO), and produce structured audit reports. The critical property: audit requires zero trust in the operator. Every verification is performed independently using cryptographic proofs and public blockchain data.

獨立第三方能否驗證所發生的一切？

整個生命週期存在的理由。稽核員可以驗證任何個別 VEO（重新計算雜湊、驗證簽名、重現 ECS、驗證 Merkle 證明）、遍歷信任鏈（從任何 VEO 跟隨血統到其根）、驗證策略合規（對照 TRO 檢查每個 VEO），並產生結構化稽核報告。關鍵特性：稽核不需要信任操作者。每個驗證都使用密碼學證明和公開的區塊鏈資料獨立執行。

4.4 — Lifecycle Invariants

4.4 — 生命週期不變量

Regardless of which profile is used, the following invariants always hold:

  1. Every stage produces a verifiable artifact. There are no "trust me" steps. If a stage was executed, its output can be independently verified.
  2. No stage requires trusting the previous stage's operator. Each artifact carries its own proof. A Stage 5 VEO can be verified without trusting the Stage 2 authorizer or the Stage 1 generator.
  3. The lifecycle is re-entrant. A VEO from one lifecycle instance can be an input to another. This enables composability: complex trust chains are built from simple, individually verifiable units.
  4. Partial lifecycles are valid. Stopping at Stage 3 is a valid, complete interaction. The system does not require all stages to produce value.
  5. Trust does not amplify. A child VEO cannot claim higher trust than its parent. Trust can only degrade or remain constant — never increase.
  6. Lifecycle metadata is append-only. Once a stage's artifact is produced, it is immutable. Corrections require a new lifecycle instance.
  7. Time moves forward. Each stage's timestamp must be equal to or later than the previous stage's. Temporal ordering is enforced and verifiable.

無論使用哪個配置，以下不變量始終成立：

  1. 每個階段都產生可驗證的成品。沒有「請相信我」的步驟。如果一個階段被執行了，其輸出可以被獨立驗證。
  2. 沒有階段需要信任前一階段的操作者。每個成品攜帶自己的證明。第五階段的 VEO 可以在不信任第二階段授權者或第一階段生成者的情況下驗證。
  3. 生命週期是可重入的。一個生命週期實例的 VEO 可以成為另一個的輸入。這實現了可組合性：複雜的信任鏈由簡單的、可個別驗證的單元構建。
  4. 部分生命週期是有效的。在第三階段停止是有效的、完整的互動。系統不需要所有階段來產生價值。
  5. 信任不會放大。子 VEO 不能聲稱比其父代更高的信任。信任只能降級或保持恆定——永遠不會增加。
  6. 生命週期元數據是僅可追加的。一旦階段的成品產出，它就是不可變的。更正需要新的生命週期實例。
  7. 時間向前移動。每個階段的時間戳必須等於或晚於前一階段的。時間順序被強制執行且可驗證。
