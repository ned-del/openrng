The Dual Naming Convention

Every patent in the family has two names. One for the patent system. One for the world.

LayerNameAudienceExample
Technical ProblemWhat can't be done todayExaminers, attorneys, engineers"Randomness can be previewed before commitment"
System PropertyWhat solving it createsPapers, talks, standards, industry"Dual Temporal Integrity"
Movement LanguageWhy it matters to the worldManifestos, keynotes, community"The Second Law of Computational Trust"

Never mix these layers. Patent examiners need problems, not movements. Conferences need properties, not claims. Manifestos need beliefs, not specifications.

The Logical Curve

The five patents form a progression — each building on the previous, each asking a deeper question about trust:

  #1   Can randomness be verified?
        │
        ▼
  #2   Can randomness be hidden until it should exist?
        │
        ▼
  #3   Can trust survive after randomness is consumed?
        │
        ▼
  #4   Can trust survive delegation?
        │
        ▼
  #5   Can trust survive time?

Patents #1–#3 answer: How to build trust.
Patents #4–#5 answer: How trust behaves across space and time.

The Meta-Framework

When is an idea ready to become a patent? When it has completed all four stages:

  Technical Problem       What can't be done today?
         │                "Trust cannot survive delegation."
         ▼
  System Property         What property does solving it create?
         │                "End-to-End Verifiable Trust"
         ▼
  Protocol / RFC          How does the protocol work?
         │                RFC-0002: Trust Continuity Protocol
         ▼
  Patent Claims           What specific inventions are claimed?
                          Method for E2E trust verification across
                          computational delegation chains

The readiness test: If you have the Problem but not the Property, it's a bug report. If you have the Property but not the Problem, it's marketing. When you have both, it's a patent.
