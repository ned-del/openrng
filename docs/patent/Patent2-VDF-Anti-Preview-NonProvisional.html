Background of the Invention

1. The Insider Preview Vulnerability in Batch Random Number Generation

[0007] In many applications — online gaming, AI agent task allocation, clinical trial randomization, financial algorithm execution, lottery systems, and autonomous decision-making — random numbers must be generated in advance, stored in pools, and served on demand at very low latency (often sub-2 milliseconds). This batch-and-pool architecture is necessary because real-time random number generation cannot meet the latency requirements of high-throughput applications.

[0008] Merkle-tree-based verification systems (such as those described in the First Application) provide an important integrity guarantee for batch random number systems: once the Merkle root hash is computed and anchored to an immutable ledger, any modification to any leaf value in the tree is cryptographically detectable. This provides post-commitment integrity — the guarantee that values were not tampered with after the batch was committed.

[0009] However, Merkle-tree verification alone does not address a distinct and equally critical vulnerability: the insider preview attack. In this attack:

  1. The system operator generates one or more candidate batches of N random values and constructs Merkle trees for each.
  2. Before anchoring any root hash to the immutable ledger, the operator previews all values in all candidate batches.
  3. The operator selects which batch to anchor, choosing a batch whose values are favorable to the operator's interests (or unfavorable to specific consumers, competitors, or counterparties).
  4. The selected batch passes all Merkle verification checks because the values were never modified after commitment — the attack does not alter values but rather selects among pre-computed alternatives.

[0010] The insider preview attack is particularly dangerous because it is undetectable by post-commitment verification alone. The Merkle tree is cryptographically intact. The blockchain anchor is genuine. All verification checks pass. The manipulation occurred in the temporal gap between value generation and commitment — a gap that Merkle verification, by design, does not cover.

2. Limitations of Existing Approaches

[0011] Hardware Random Number Generators (HRNGs). HRNGs produce entropy from physical processes (thermal noise, radioactive decay, quantum phenomena). While the entropy quality may be excellent, HRNGs provide no mechanism to prove that the operator did not preview, buffer, filter, or selectively discard the output before incorporating it into a batch. The physical entropy generation is not time-locked; values are available instantly upon generation, enabling immediate preview.

[0012] Pseudo-Random Number Generators (PRNGs) and Cryptographically Secure PRNGs (CSPRNGs). PRNGs and CSPRNGs are deterministic given a seed. If the seed is known — or can be determined — before batch generation, the entire batch is immediately computable and therefore previewable. Even CSPRNGs that use high-quality seeds (e.g., from /dev/urandom) do not prevent the operator from computing and previewing the full batch instantaneously, because the computation is fast and non-sequential.

[0013] Verifiable Random Functions (VRFs). VRFs, such as those described in U.S. Patent No. 12,401,532 B2 (Chainlink, "Method and Apparatus for Producing Verifiable Randomness Within a Decentralized Computing Network"), prove that a specific output was correctly derived from a specific input using a specific secret key. However, VRFs have several limitations in the context of anti-preview batch generation:

  1. VRFs are typically designed for single-value generation, not batch generation of thousands of values for pool-based serving.
  2. The holder of the VRF secret key can compute the VRF output for any input instantaneously, enabling preview of future outputs given anticipated future inputs.
  3. VRFs do not inherently introduce a time-lock: the VRF computation is fast (typically sub-millisecond), allowing rapid generation and evaluation of multiple candidate outputs.
  4. VRFs require an on-chain callback mechanism for verification, preventing offline or air-gapped verification.
  5. VRFs typically use a single entropy source (the secret key holder's computation), not multi-source aggregation.

[0014] Commit-Reveal Schemes. Traditional commit-reveal protocols require interactive participation from multiple parties: one party commits to a value (publishing a hash), and later reveals the value. While this prevents single-party manipulation, commit-reveal schemes require real-time coordination, introduce latency equal to the reveal delay, require all parties to be online simultaneously, and do not scale to batch generation of tens of thousands of values for pool-based sub-millisecond serving.

[0015] Trusted Execution Environments (TEEs). TEEs (Intel SGX, ARM TrustZone, AWS Nitro Enclaves) provide hardware-isolated execution that prevents the host from inspecting computation. However, TEEs prove execution isolation, not temporal unpredictability. A TEE can faithfully execute a batch generation algorithm, but the TEE attestation does not prove that the values were unknowable before a specific time — only that the execution environment was not tampered with. TEE-based approaches also introduce hardware dependency and supply-chain trust assumptions.

[0016] Public Randomness Beacons (e.g., drand, NIST Randomness Beacon). Public beacons produce verifiable random values at regular intervals. However, beacons produce single values per round (not batches), operate at fixed intervals (typically 3-30 seconds, too slow for per-request serving), and do not address the batch-to-pool pipeline required for sub-millisecond serving.

3. Verifiable Delay Functions — Prior Art and Distinction

[0017] Verifiable Delay Functions (VDFs) are a class of cryptographic primitive formally defined by Boneh, Bonneau, Bünz, and Fisch ("Verifiable Delay Functions," IACR ePrint 2018/601, 2018). A VDF takes a prescribed amount of sequential computation time T to evaluate, even on a parallel computer, and produces an output accompanied by a proof that can be efficiently verified (typically in O(log T) time).

[0018] VDFs have been proposed and implemented in several contexts:

  1. Boneh et al. (2018): Theoretical definition of VDFs, including Wesolowski and Pietrzak constructions based on repeated squaring in groups of unknown order.
  2. Tezos (Nomadic Labs, 2022): Integration of VDFs into the Kathmandu protocol upgrade for improving random seed generation for block proposer selection in proof-of-stake consensus.
  3. StarkWare VeeDo: A VDF computation service with STARK proof verification on Ethereum, demonstrating VDF + on-chain verification.
  4. Ethereum Research: Proposals for VDF-based randomness in Ethereum's consensus layer.

[0019] While the VDF primitive itself is known, no prior art describes the specific combination of innovations in the present invention: (i) integration of VDFs into a batch random number generation pipeline producing tens of thousands of values from a single VDF output, (ii) combination of VDF anti-preview with Merkle anti-tamper to provide dual temporal integrity, (iii) asynchronous blockchain anchoring decoupled from token availability, (iv) multi-worker VDF computation with priority-based scheduling, (v) shared and dedicated pool architectures with automatic refill, (vi) hierarchical entropy source fallback with transparent provenance recording, and (vii) integration with distributed randomness beacons for publicly verifiable VDF input seeds.

2. VDF-Protected Seed Generation Pipeline

[0024] The core innovation of the present invention is the VDF-protected seed generation pipeline, which creates a temporal barrier ensuring that no party can preview the random values before a minimum sequential computation time T has elapsed. The pipeline comprises the following steps:

2.1 Entropy Acquisition

[0025] The system obtains entropy from one or more independent sources arranged in a hierarchical fallback architecture (FIG. 6). In the preferred embodiment, the hierarchy is:

[0026] Tier 1 — Distributed Randomness Beacon. The preferred entropy source is a distributed randomness beacon, such as the drand network operated by the League of Entropy (comprising Cloudflare, Protocol Labs, academic institutions, and other independent operators). The drand beacon produces a new random value approximately every 3 seconds. Each beacon round provides: (a) a round number, (b) a randomness value (32 bytes, hexadecimal-encoded), and (c) a BLS threshold signature. The system verifies the beacon's integrity by confirming that SHA-256(signature) equals the published randomness value. In alternative embodiments, other distributed beacons may be used, including the NIST Randomness Beacon, Ethereum's RANDAO, or any future publicly verifiable randomness service.

[0027] Tier 2 — Blockchain Block Entropy. If the distributed beacon is unavailable (network failure, timeout exceeding a configurable threshold, integrity check failure, or consecutive failures exceeding a configurable maximum), the system obtains entropy from one or more blockchain block hashes. Block hashes from proof-of-work chains (e.g., Bitcoin) provide entropy derived from the computational effort of the mining network. Block hashes from proof-of-stake chains (e.g., Ethereum, Polygon) provide entropy derived from validator consensus. Multiple block hashes from independent chains may be combined for increased source diversity.

[0028] Tier 3 — Local Cryptographic Entropy. As a final fallback, the system uses operating system cryptographic randomness (e.g., /dev/urandom, crypto.randomBytes(), or equivalent) combined with system-specific entropy (timestamps, process identifiers, hardware counters). This tier provides the lowest source diversity but maintains the anti-preview guarantee because the VDF computation time is independent of the entropy source.

[0029] Provenance Recording. The entropy source tier used for each batch is recorded in the batch metadata. This transparent provenance enables consumers and auditors to distinguish between batches sourced from publicly verifiable beacons (Tier 1), blockchain-derived entropy (Tier 2), and locally-sourced entropy (Tier 3). There are no silent fallbacks — every degradation in source quality is visible in the metadata.

2.2 Seed Construction

[0030] The system constructs a VDF input seed by combining entropy from the selected tier with additional entropy contributions using one or more cryptographic hash functions. In the preferred embodiment:

vdfSeed = H(blockParam ‖ randomEntropy ‖ additionalEntropy)

[0031] where H is a cryptographic hash function (SHA-256 in the preferred embodiment; SHA-3, BLAKE2, BLAKE3, or other cryptographic hash functions in alternative embodiments), blockParam is derived from a blockchain block hash, randomEntropy is the beacon or local entropy value, and additionalEntropy is optional additional randomness (e.g., from a hardware random number generator). The concatenation operator ‖ denotes bitstring concatenation.

[0032] The use of multiple entropy contributions ensures that no single source controls the VDF input seed. Even if one entropy source is compromised or predictable, the remaining sources contribute unpredictability to the seed.

2.3 VDF Computation

[0033] The VDF input seed is processed through a Verifiable Delay Function (VDF) requiring a minimum sequential computation time of T seconds. The VDF computation is the anti-preview barrier: no party — including the system operator, the entropy source operators, and any external attacker — can determine the VDF output before T seconds of sequential computation have elapsed.

[0034] In the preferred embodiment, the VDF is implemented as an iterative sequential hash chain:

h[0] = H(seed ‖ "0")
h[1] = H(h[0] ‖ "1")
h[2] = H(h[1] ‖ "2")
...
h[K] = H(h[K-1] ‖ "K") = vdfOutput

[0035] where K = T × C (T being the target computation time in seconds, C being the number of hash operations per second for the deployment hardware — default C = 10,000 in the preferred embodiment). Each iteration depends on the previous iteration's output, creating a strict sequential dependency that prevents parallelization. Even with unlimited parallel processors, the computation cannot complete in fewer than K sequential hash operations.

[0036] The VDF produces two outputs: (a) the VDF output value (the final hash h[K]), and (b) a VDF proof enabling efficient verification. In the preferred embodiment, the proof is computed as H(vdfOutput ‖ seed). In alternative embodiments, the VDF may be implemented using:

  1. Wesolowski VDF: Repeated squaring in a group of unknown order (e.g., RSA group), with a succinct proof verifiable in O(log T) time.
  2. Pietrzak VDF: An alternative construction with a larger proof size but potentially faster verification.
  3. Sloth function: Modular square root computation with inherent sequentiality.
  4. Any future VDF construction that provides the sequential computation property and efficient verifiability.

[0037] The configurable parameter T allows the system to balance anti-preview security against batch generation throughput. Higher T values provide stronger anti-preview guarantees (longer windows during which values are unknowable) but reduce the rate at which new batches can be generated. In the preferred embodiment, T defaults to 4 seconds, which is sufficient to prevent real-time preview attacks while enabling multiple batches per minute via multi-worker parallelism.

3. Deterministic Batch Derivation

[0040] From the VDF output, the system deterministically generates N random values through the following process:

[0041] Step 1 — RN Generation Parameter. The VDF output is combined with the block parameter to produce the RN (Random Number) Generation Parameter:

rnGenParam = H(vdfOutput ‖ blockParam)

[0042] Step 2 — Leaf Hash Generation. For each leaf index i (0 ≤ i < N), the system generates a unique node identifier and computes a leaf hash:

nodeId[i] = formatIdentifier(i, batchId)
leafHash[i] = H(rnGenParam ‖ nodeId[i])

[0043] Step 3 — Numeric Value Derivation. Each leaf hash is converted to a numeric random value by extracting a prefix of the hash and mapping it to the target range. In the preferred embodiment, the first 8 hexadecimal characters are extracted and divided by 0xFFFFFFFF, producing a uniformly distributed floating-point value in the range [0, 1). For integer ranges, the value is scaled and truncated: intValue = floor(floatValue × (max - min + 1)) + min.

[0044] The batch size N is configurable. In the preferred embodiment, N = 65,536 (2^16). In alternative embodiments, N may range from 1,024 (2^10) for low-throughput applications to 1,048,576 (2^20) or more for high-throughput deployments. The derivation is fully deterministic: given the same rnGenParam and batch metadata, any party can independently regenerate the identical batch.

4. Merkle Tree Construction and Verification

[0045] The N leaf hashes are organized into a binary Merkle tree. Adjacent leaves are concatenated and hashed pairwise to produce parent nodes, recursively until a single root hash is obtained. When the number of leaves at any level is odd, the last leaf is duplicated to form a complete pair.

[0046] Each individual token can be verified against the Merkle root using a proof path of O(log₂ N) hash comparisons. For N = 65,536, this requires 16 hash comparisons — computationally trivial. The verification algorithm traverses the proof path, combining the leaf hash with sibling hashes at each level, and confirms that the final computed value equals the published root hash.

[0047] Verification requires no cooperation from the system operator. Any party with access to: (a) the leaf hash, (b) the proof path, (c) the on-chain root hash, and (d) optionally the VDF provenance metadata, can independently verify the token. This verification can be performed offline, air-gapped, and at any future time.

5. Asynchronous Anchoring Architecture

[0048] A key architectural innovation is the decoupling of token availability from immutable ledger confirmation. In conventional systems, tokens cannot be served until the blockchain transaction confirming the Merkle root is finalized — introducing latency of seconds to minutes. The present invention decouples these operations:

  1. Immediate injection: Tokens are injected into the serving pool immediately upon batch derivation, before blockchain anchoring. The anti-preview guarantee has already been provided by the VDF computation; the Merkle tree provides anti-tamper; the blockchain anchor adds temporal proof and immutability but is not required for the anti-preview or anti-tamper properties.
  2. Asynchronous anchoring: The Merkle root is submitted to the immutable ledger (blockchain smart contract, in the preferred embodiment) in a background process with configurable retry logic (default: 3 attempts with exponential backoff).
  3. Status tracking: Each token carries an anchor status field that transitions through states: "pending" (submitted but not confirmed), "confirmed" (transaction confirmed at specific block), "failed" (all retry attempts exhausted). The status is queryable by consumers.
  4. Retroactive upgrade: Upon anchor confirmation, all tokens in the batch are retroactively upgradeable to the highest verification level (anchored verification) without regeneration.

[0049] In the preferred embodiment, the immutable ledger is a smart contract on the Polygon blockchain (contract address: 0xA79E149C35Ad47Ed270Bf4b16B80170eBF7B88F8 on Polygon Amoy testnet). In alternative embodiments, the immutable ledger may be any blockchain (Ethereum, Bitcoin via OP_RETURN, Solana, etc.), any append-only ledger, any transparency log, or any distributed timestamping system that provides immutability and public verifiability.

Consumers access the shared pool with per-consumer rate limiting (configurable; default: 100 requests per minute). Refill is triggered at configurable thresholds: 50% for standard refill (one batch), 20% for critical multi-batch emergency refill (up to 3 concurrent batches).

[0055] Dedicated Pool. Per-consumer pools provide isolation for verticals requiring independent audit trails (e.g., regulated gaming, clinical trials). Each consumer has an independent pool with independent refill triggers, batch history, and audit trail. Dedicated pools are pre-warmed with one batch upon consumer registration.

8. Token Serving and Proof Delivery

[0056] Referring to FIG. 7, when a consumer requests one or more random tokens, the system:

  1. Checks rate limits (for shared pool consumers).
  2. Consumes the requested number of tokens from the appropriate pool (shared or dedicated) using a sequential pointer (O(1) retrieval).
  3. For each consumed token, generates a Merkle proof path from the token's leaf hash to the batch's Merkle root.
  4. Packages the response including: the random value, the leaf hash, the node identifier, the batch identifier, the complete Merkle proof path, the Merkle root, the VDF provenance metadata (entropy source, VDF computation time, VDF proof), and the blockchain anchor reference (transaction hash and block number, or "pending" if not yet confirmed).
  5. Records the consumption event (consumer identifier, timestamp, token identifiers).
  6. Checks pool depth and triggers refill if below threshold.

[0057] The entire serving process operates at sub-2-millisecond latency because all data is served from in-memory structures. No VDF computation, no blockchain query, and no external service call occurs during serving — those operations happened during batch generation (VDF) and background anchoring (blockchain).

9. Independent Verification

[0058] Any party may independently verify a served token by performing the following checks, none of which requires cooperation from the system operator:

  1. Integrity check: Recompute H(rnGenParam ‖ nodeId) and confirm it equals the claimed leaf hash.
  2. Merkle proof check: Traverse the proof path, combining the leaf hash with sibling hashes at each level, and confirm the computed root equals the published Merkle root.
  3. Blockchain anchor check: Query the on-chain smart contract to confirm the published Merkle root is recorded at the claimed transaction hash and block number.
  4. VDF provenance check: Verify the VDF proof against the VDF seed, confirming the VDF computation was performed correctly.
  5. Entropy source check: If the entropy source was a distributed beacon (Tier 1), independently fetch the beacon round and verify its BLS signature against the published chain public key.

[0059] These checks can be performed offline (except the blockchain anchor check, which requires network access or a local chain node), at any future time, and without any interaction with the system operator.

10. Alternative Embodiments

10.1 Alternative VDF Constructions

[0060] While the preferred embodiment uses an iterative sequential hash chain, alternative embodiments may employ any VDF construction that provides the sequential computation property, including but not limited to: Wesolowski VDF (repeated squaring in RSA groups), Pietrzak VDF (halving protocol), sloth functions (modular square roots), or any future VDF construction. The choice of VDF construction does not affect the system architecture or the anti-preview guarantee — it affects only the verification efficiency and proof size.

10.2 Alternative Hash Functions

[0061] While the preferred embodiment uses SHA-256, alternative embodiments may use any cryptographic hash function including SHA-3 (Keccak), BLAKE2, BLAKE3, or future hash functions. The system architecture is hash-function-agnostic; the hash function is a configurable parameter.

10.3 Alternative Immutable Ledgers

[0062] While the preferred embodiment uses a Polygon blockchain smart contract, alternative embodiments may anchor Merkle roots to any immutable verification substrate including: public blockchains (Ethereum, Bitcoin, Solana, etc.), consortium chains, private chains, append-only distributed ledgers, certificate transparency logs, notarization networks, or distributed timestamping systems. The system architecture requires only that the substrate provides immutability (the recorded root cannot be retroactively altered) and public verifiability (any party can confirm the root was recorded at a specific time).

10.4 Alternative Entropy Sources

[0063] While the preferred embodiment uses drand as the primary entropy source, alternative embodiments may use any publicly verifiable entropy source including: NIST Randomness Beacon, Ethereum RANDAO, quantum random number generators (QRNGs), multiple independent randomness beacons combined, satellite-based entropy, or federated entropy networks. The system architecture is entropy-source-agnostic; new sources are added by implementing a source adapter.

10.5 Integration with Verifiable Entropy Objects

[0064] In an embodiment integrated with the Third Application (VEO patent), the VDF-protected batch generation pipeline serves as the Trust Source layer (Layer 1) of the Computational Trust Infrastructure. Each batch's provenance metadata — entropy source, VDF computation parameters, VDF proof, Merkle root, and blockchain anchor — is encapsulated in a Verifiable Entropy Object (VEO-1), providing a standardized, self-describing, independently verifiable trust evidence package. The VEO's Entropy Confidence Score (ECS) reflects the VDF-protected batch's trust properties: source diversity (multiple entropy contributions to the VDF seed), manipulation resistance (VDF temporal barrier), and verification success (VDF proof validity and beacon signature verification).

10.6 AI Agent Decision Attestation

[0065] In an embodiment for AI agent systems, VDF-protected random tokens are used to back autonomous agent decisions — tool selection, task routing, resource allocation, response sampling. Each decision is linked to the specific token (and its VDF provenance) that influenced it, creating an auditable decision trail. The VDF anti-preview guarantee ensures that the agent (or its operator) could not have previewed the random input to a decision before the decision was made, preventing selective manipulation of agent behavior.

10.7 Regulated Gaming

[0066] In an embodiment for regulated gaming (casinos, lotteries, online gambling), VDF-protected batches serve as the random number source for game outcomes. The dual temporal integrity guarantee — VDF anti-preview plus Merkle anti-tamper — satisfies the requirements of gaming regulators (GLI-19, GLI-33, BMM, eCOGRA) for provably fair randomness. Dedicated pools per operator provide independent audit trails. The VDF computation time T is configurable per regulatory jurisdiction.

10.8 On-Demand Generation Fallback

[0067] In an embodiment handling pool exhaustion, the system generates random values on-demand when the pool is depleted. On-demand generation bypasses the full VDF pipeline (using direct cryptographic randomness) to avoid latency, but flags the tokens with reduced provenance metadata indicating on-demand generation. Simultaneously, the system triggers emergency pool refill via the VDF pipeline. This fallback ensures zero downtime while transparently indicating that on-demand tokens do not carry the full VDF anti-preview guarantee.

Abstract of the Disclosure

A method and system for generating batches of random numbers with verifiable anti-preview guarantees using Verifiable Delay Functions (VDFs). The system obtains entropy from independent sources including distributed randomness beacons and blockchain block hashes through a hierarchical fallback architecture with transparent provenance recording. The entropy is processed through a VDF requiring a minimum sequential computation time T, and deterministically derives a batch of N random values from the VDF output. The batch is organized as a hash tree (Merkle tree) whose root hash is recorded on an immutable verification substrate (blockchain). The VDF computation time provides an anti-preview guarantee: no party, including the system operator, can determine the batch values before the VDF computation completes. The hash tree provides an anti-tamper guarantee: no value can be altered after the root is computed. Together, these guarantees provide dual temporal integrity — values are neither knowable before generation nor alterable after generation. The system supports sub-2-millisecond token serving from pre-generated pools, multi-worker VDF computation for parallel batch generation, shared and dedicated pool architectures with automatic threshold-based refill, asynchronous blockchain anchoring decoupled from token availability, and integration with structured verifiable entropy object standards.
