Patent #1

Verifiable Randomness

Technical Problem: Random values cannot be independently verified
System Property: Verifiable Randomness
Status: ✅ APPROVED
Title: Method and System for Gaming Random Number Generation
Patentability 9/10
Commercial Value 7/10
Defensibility 7/10

The Problem

When a casino pays out $2 million on a jackpot, regulators need to verify the winning number was genuinely random. Current systems produce audit logs written by the same system they audit. The randomness itself carries no proof.

The Invention

A system that generates random numbers as Verifiable Entropy Objects (VEOs) — cryptographic containers carrying entropy payload, source provenance, confidence metrics, provider signatures, and blockchain anchors. Any party can independently verify a VEO without trusting the provider.

Core Innovation

The VEO-1 standard — transforms randomness from a disposable value into a verifiable digital object with five layers: Payload, Provenance, Confidence, Proof, and Anchor.

Key Technical Components

  • VDF (Verifiable Delay Functions) for time-locked entropy generation
  • drand public randomness beacon for independent entropy sourcing
  • Merkle tree batching for efficient verification of large token pools
  • Polygon blockchain anchoring for immutable proof
  • Entropy Confidence Score (ECS): 0–1000 quality metric across 6 dimensions

⚠ Risk Assessment

Approved and foundational — strongest legal footing in the portfolio. However, Chainlink's US12401532B2 disclosure explicitly mentions VDF combination. While their claims are specific to distributed VRF oracle networks, the disclosure creates prior art exposure that requires a formal claim-by-claim analysis to quantify. The centralized-vs-distributed architectural distinction is the primary moat.

Patent #2

Dual Temporal Integrity

Technical Problem: Random values can be previewed before commitment
System Property: Dual Temporal Integrity
Status: 📋 Ready to file (Provisional + Non-Provisional drafted)
Patentability 8/10
Commercial Value 8/10
Defensibility 9/10

The Problem

The "Insider Preview Attack" — in traditional RNG, the system generating randomness can preview values before they're committed. An operator, insider, or compromised system could select favorable outcomes.

The Invention

A dual-phase temporal commitment scheme where:

  1. Random values are generated and cryptographically committed (Merkle root anchored on-chain) BEFORE any request arrives
  2. Values are revealed only upon request, with the pre-commitment proving they existed before the request

Core Innovation

Separating the generation timeline from the consumption timeline through VDF-enforced time delays + blockchain anchoring, making preview attacks mathematically impossible.

Differentiation from Chainlink VRF

Chainlink uses request-response (randomness generated AFTER request). OpenRNG pre-generates pools of committed values — fundamentally different temporal model.

⚠ Risk Assessment

Cleanest claims in the portfolio. Strongest §103 defense via the dual-path pipeline — pre-generation temporal model is fundamentally different from Chainlink's request-response architecture. The pre-generated pool with blockchain-anchored Merkle roots is structurally distinct from every existing verifiable randomness system. Examiner will likely cite general commitment schemes (Blum 1981); defense must emphasize the pool architecture, not the commitment concept.

Patent #3

Portable Trust Assertion

Technical Problem: Trust disappears after randomness is consumed
System Property: Portable Trust Assertion
Status: 📋 Ready to file (Provisional + Non-Provisional drafted)
Patentability 7/10
Commercial Value 7/10
Defensibility 7/10

The Problem

Once a random number is consumed, the trust evidence that backed it is discarded. A slot machine uses a random number, records the outcome, but the cryptographic proof of that number's integrity is lost. Trust is disposable.

The Invention

Trust Assertions that persist after entropy consumption — portable, verifiable records that any party can independently verify at any future time. The VEO becomes a permanent trust artifact, not a disposable value.

Core Innovation

The concept of "entropy receipts" — each consumption generates a portable attestation record that carries full provenance, quality scores, and verification paths. Trust becomes a first-class, persistent data object.

Key Continuation Anchors

This patent's non-provisional specification includes the Five-Layer Trust Stack and Decision Attestation Records — securing earliest priority dates for Patents #4 and #5.

⚠ Risk Assessment

Solid but carries the broadest claims of the gaming cluster. The "entropy receipt" concept is novel but must be aggressively distinguished from standard digital receipts, certificates, and blockchain-based proof-of-existence systems. A hostile examiner could characterize this as "logging with cryptographic signatures." The Five-Layer Trust Stack and ECS quality metrics are the key differentiators — without them, the claims collapse into existing attestation/notarization prior art.

Patent #4

End-to-End Verifiable Trust

Technical Problem: Computational trust cannot survive delegation
System Property: End-to-End Verifiable Trust
Status: 🔬 Problem defined, drafts completed
Patentability: 70–80%
Patentability 6/10
Commercial Value 8/10
Defensibility 5/10

The Problem

When a trusted computation delegates work to another agent, the trust breaks. The delegation boundary destroys three things:

  1. Provenance linkage — delegated agent gets task but not trust artifact
  2. Quality continuity — no mechanism to detect trust quality degradation through delegation
  3. Audit continuity — auditor cannot trace back to original trust assertion

Why Existing Approaches Fail

  • Task queues (Celery, SQS) deliver tasks, not trust
  • API auth (OAuth, JWT) proves WHO delegated, not WHY trustworthy
  • Distributed tracing (OpenTelemetry) traces execution path, not trust path
  • Blockchain logs record THAT delegation occurred, not the trust that AUTHORIZED it
  • PKI proves identity delegation, not computational trust delegation

The Invention

A system for preserving computational trust across delegation boundaries by cryptographically linking child trust assertions to parent assertions through lineage records that enforce non-amplification, record degradation transparently, and enable independent verification of the complete trust chain from any node to root — without cooperation from intermediate nodes.

Target Market

Multi-agent AI systems (OpenAI, Anthropic), cross-organizational computation, regulatory compliance chains.

⚠ Risk Assessment

Weakest link in the portfolio. Independent claims are currently too narrow — over-specified to DAG mechanics, making the patent easy to design around. Needs an abstraction pass to broaden independent claims while keeping DAG as one embodiment. Carries the highest Alice §101 risk: "preserving trust across delegation boundaries" sounds abstract. Central tension: broadening claims for defensibility increases §101 exposure. The DAG specificity is actually a §101 safe harbor — losing it requires careful reanchoring to concrete technical operations.

Action Needed: Add broader independent claim beyond DAG mechanics — let DAG be one embodiment, not the only claim.
Patent #5

Continuous Trust Health

Technical Problem: Trust quality silently degrades during operation
System Property: Continuous Trust Health
Status: 🔬 Problem defined, drafts completed
Patentability: 80–90% (highest of all five)
Patentability 8/10
Commercial Value 9/10
Defensibility 7/10

The Problem — Trust Decay

Trust infrastructure doesn't fail catastrophically — it rots. Entropy sources silently degrade. Quality scores drift. Trust chains grow deeper than policy allows. Anchoring latency creeps up. Source diversity narrows. None triggers an error. Each is individually small. Collectively, they transform a AAA-grade system into LOW-grade — without anyone noticing.

The Battery Analogy

Trust Decay is to trust infrastructure what battery drain is to mobile devices. A fully charged battery (AAA trust) slowly loses charge. The drain is not an error — it is physics. Without monitoring, the user doesn't know until the device dies. Trust infrastructure needs the same: a trust health meter, decay detection, and active maintenance.

Why Existing Tools Fail

  • SIEM (Splunk) monitors security events, not trust quality
  • APM (Datadog) monitors performance — system is fast while trust silently degrades
  • GRC platforms require manual periodic checks — Trust Decay is continuous
  • Alerting systems catch threshold breaches — Trust Decay is sub-threshold (many small drifts)

The Invention

A system for detecting, quantifying, and counteracting continuous trust decay through:

  • Multi-dimensional trust health monitoring
  • Anomaly detection against trust-specific metrics (not security/performance)
  • Automated corrective response (source rebalancing, policy tightening)
  • Continuous machine-readable compliance attestation
  • Predictive trust quality analytics forecasting decay before critical thresholds

Commercial Value

Highest of all five patents. "Trust Health" creates a new monitoring category. Every compliance-heavy industry needs this. Google/Microsoft interested for governance/compliance.

⚠ Risk Assessment

Highest novelty and commercial value in the portfolio. "Trust Health" as a new category is a strong framing — no existing patent or academic paper treats trust quality as a time-varying signal requiring continuous monitoring. Key threats: Cyberscope's Aug 2025 patent filing for "AI-Optimized Blockchain Trust Scoring Platform" and Yaslioglu's 2025 arXiv paper on Bayesian time-decay trust in blockchain. Both are distinguishable (Cyberscope = smart contract audit; Yaslioglu = reputation trust in P2P networks), but expect examiners to cite them.

Section V

Red Team Summary

The three highest-probability attack vectors opposing counsel would deploy against this portfolio, and the consolidated defense playbook.

Top 3 Attack Vectors

Attack #1 — §103 Obviousness Combination

Target: Patents #1, #2, #3

Argument: "Chainlink VRF (US12401532B2) + Amazon Merkle trees (US10291408B2) = obvious combination. VDF is known (Boneh 2018). Blockchain anchoring is known (Bitcoin 2008). The combination of these known techniques to verify randomness is obvious to one skilled in the art."

Threat level: HIGH — This is the most likely attack and will be attempted during prosecution.

Attack #2 — Alice Corp §101 Abstract Idea

Target: Patents #4, #5

Argument: "Trust management is an abstract concept. 'Preserving trust across delegation boundaries' (#4) and 'monitoring trust quality over time' (#5) are mental processes that humans perform. Implementing them on a computer does not transform them into patent-eligible subject matter."

Threat level: HIGH — §101 rejections are increasingly common for software patents. Patent #4 is most exposed.

Attack #3 — §102 Anticipation via Academic Prior Art

Target: Patents #1, #2

Argument: "VDF academic papers (Boneh et al. 2018) + commit-reveal schemes (Blum 1981) anticipate the core claims. The specific combination of pre-generation with VDF-derived seeds was foreseeable from the academic literature."

Threat level: MEDIUM — Anticipation requires a single reference; the combination argument is weaker here than §103.

Defense Playbook

✅ Primary Defense: Hardware VDF Computational Bottleneck

  • VDF computation is inherently sequential — cannot be parallelized or shortcut
  • This hardware constraint is the §101 safe harbor for #4/#5: the claims are anchored to a physical computational limitation, not an abstract idea
  • No human can perform VDF computation, Merkle tree verification, or real-time entropy source rebalancing — these are inherently computational processes

✅ Secondary Defense: Dual-Path Pipeline

  • Sub-2ms in-memory token serving + slow background Merkle anchoring = non-obvious engineering tradeoff
  • No prior art system separates the serving path (in-memory, instant) from the verification path (blockchain, delayed)
  • This architectural choice is not suggested by any combination of Chainlink + Amazon + academic papers

✅ Tertiary Defense: Pre-Generation Pool Architecture

  • Pre-generation pool model is architecturally distinct from all prior art request-response models
  • Chainlink generates randomness AFTER request; OpenRNG generates BEFORE — fundamentally different temporal architecture
  • Batches of 65,536 VDF-derived tokens committed as a single Merkle root — no existing system proposes this
